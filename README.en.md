## Project Overview

This project addresses the urban **cruising-for-parking** problem in a Central Business District (CBD) by combining **SUMO** (Simulation of Urban MObility) microscopic traffic simulation with a **PostgreSQL** real-time reservation database.

**Two scenarios are compared:**

| Scenario | Description |
|----------|-------------|
| **A — Baseline** | Vehicles blindly cruise the entire road network searching for vacant spots. No reservation system. |
| **B — Smart Booking & Pricing** | Vehicles are assigned optimal parking spots at spawn time via a cost function combining distance and dynamic surge pricing. |

**Core capabilities:**

- **Parking Schema** — Maps SUMO parking areas into a relational database with real-time state tracking.
- **Cruising Detection** — Monitors each vehicle via TraCI to record search time, distance driven, and fuel consumption.
- **Reservation Engine** — Assigns and reserves available spots for incoming vehicles based on SQL state queries.
- **Dynamic Pricing** — Implements surge pricing: occupancy > 70% triggers 1.5× base price; > 90% triggers 2×.
- **Performance Dashboard** — A Streamlit-based web dashboard comparing key metrics across both scenarios.

## Database Design

The system uses **PostgreSQL** for real-time state synchronization. Connection parameters are configured in `.env` at the project root. The schema is auto-generated by `scripts/generate_parking.py` and written to `configs/schema.sql`.

### Table: `Parking_Spots`

Records the real-time state of all off-street and on-street parking spots.

| Column | Type | Description |
|--------|------|-------------|
| `spot_id` | `VARCHAR(50)` | Primary key; unique spot identifier (e.g. `off_street_0`, `on_street_42`) |
| `edge_id` | `VARCHAR(50)` | The road segment (edge) the spot belongs to |
| `spot_type` | `ENUM('on-street', 'off-street')` | Category: roadside or parking lot |
| `capacity` | `INT` | Maximum number of vehicles the spot can hold |
| `occupied` | `INT` | Currently occupied / reserved count (default 0) |
| `base_price` | `DECIMAL(5,2)` | Base parking fee |
| `current_price` | `DECIMAL(5,2)` | Surge-adjusted parking fee (updated in real-time by Scenario B) |

### Table: `Cruising_Logs`

Records each vehicle's parking search lifecycle and environmental cost.

| Column | Type | Description |
|--------|------|-------------|
| `log_id` | `SERIAL` | Primary key; auto-increment |
| `vehicle_id` | `VARCHAR(50)` | Vehicle identifier |
| `scenario` | `VARCHAR(20)` | Scenario name (`Baseline` or `Smart_Booking_Priced`) |
| `search_time_sec` | `FLOAT` | Time spent searching for parking (seconds) |
| `cruising_distance_m` | `FLOAT` | Total distance driven while searching (meters) |
| `final_spot_id` | `VARCHAR(50)` | Spot where the vehicle ultimately parked (NULL if failed) |
| `total_fuel_mg` | `FLOAT` | Total fuel consumed during search (milligrams) |
| `created_at` | `TIMESTAMP` | Auto-generated log timestamp |

## Getting Started

### Prerequisites

- **Python 3.10** (recommended: use `uv` for virtual environment management)
- **PostgreSQL** running locally or remotely
- **SUMO** (Simulation of Urban MObility) installed and added to system `PATH`; `SUMO_HOME` environment variable configured
- **VS Code** (recommended). Add the following to `.vscode/settings.json` for Pylance support:

```json
{
    "python.analysis.extraPaths": ["${workspaceFolder}/scripts"]
}
```

### 1. Clone & Install Dependencies

```bash
git clone https://github.com/bulong0211/dbspre.git
cd dbspre

# Using uv (recommended)
uv sync

# Or using pip
pip install -r requirements.txt
```

### 2. Configure Database

Copy the environment template and fill in your PostgreSQL credentials:

```bash
cp .env.example .env
```

Edit `.env`:

```env
DB_NAME=smart_parking
DB_USER=postgres
DB_PASSWORD=your_password_here
DB_HOST=localhost
DB_PORT=5432
```

> **Note:** Create the database in PostgreSQL first (e.g., `CREATE DATABASE smart_parking;`).

### 3. Prepare Simulation Environment

If network and parking files already exist in `configs/`, skip this step. Otherwise, run the one-click preparation script:

```bash
uv run scripts/prepare_simulation.py
```

This sequentially executes: network generation → parking generation → traffic generation → database initialization.

### 4. Run Simulations

```bash
# Scenario A — Baseline (blind cruising, clears all historical logs on start)
uv run scripts/run_scenario_A_baseline.py

# Scenario B — Smart Booking & Dynamic Pricing
uv run scripts/run_scenario_B_smart.py
```

Each simulation runs for up to 7,200 seconds (2 hours) or until all 2,500 vehicles are processed. Real-time metrics are displayed in a matplotlib window, and results are written to PostgreSQL.

### 5. View Dashboard

Launch the Streamlit performance comparison dashboard:

```bash
uv run streamlit run scripts/run_dashboard.py
```

Open `http://localhost:8501` in your browser to compare success rates, search times, fuel consumption, and cruising distances between the two scenarios.

## Software Operation

### Simulation Flow

Both scenarios follow the same high-level loop integrated with SUMO via TraCI:

```
1. Reset database state
2. Connect to PostgreSQL
3. Load parking spot data
4. Start SUMO with the demo configuration
5. While simulation has active vehicles and time < 7200s:
   a. Advance one simulation step
   b. Update GUI camera tracking
   c. Process newly departed vehicles (assign spots / set routes)
   d. Process active driving vehicles (update metrics, detect parking, handle timeouts)
   e. Refresh the real-time plotter
   f. Check for early completion (all 2500 vehicles processed)
   g. Sync spot occupancy and prices to database every 60s
6. Final database sync
7. Close SUMO, plotter, and database connection
```

### Key Differences Between Scenarios

| Aspect | Scenario A | Scenario B |
|--------|-----------|-----------|
| Spot assignment | None (vehicles find spots by street scanning) | Optimal spot assigned at spawn via cost function |
| Navigation | Random edge rerouting | Direct route to assigned spot |
| Dynamic pricing | Not used | Surge pricing updated every step |
| Driving behavior | `setSpeedFactor(0.4)`, `setImperfection(0.9)` | Same parameters |
| Timeout | 120s target timeout with reroute | 120s target timeout with teleport |
| Parking logic | `parking_logic.py` (scan, try_park, reroute) | `changeTarget` + `setParkingAreaStop` |

## Module Reference

### `core/config.py` — Configuration Constants and Paths

Defines all tunable simulation parameters.

| Constant | Value | Description |
|----------|-------|-------------|
| `CONFIG_DIR` | `Path(...)/configs` | Path to SUMO config files |
| `SIMULATION_DURATION_LIMIT` | `7200` | Maximum simulation time (seconds) |
| `TOTAL_VEHICLES_TARGET` | `2500` | Total vehicles to process |
| `PARKING_DURATION` | `7200` | Duration a vehicle stays parked once it finds a spot |
| `SIGHT_DISTANCE` | `80.0` | Forward visibility distance for street scanning (meters) |
| `SPOT_STOP_MARGIN` | `3.0` | Minimum distance ahead to consider a spot reachable |
| `ROUTE_EXHAUSTION_MARGIN` | `5` | Edges remaining before triggering reroute |
| `INTERSECTION_LOOKAHEAD` | `40.0` | Distance from intersection to scan crossing roads |
| `TARGET_TIMEOUT` | `120` | Max seconds locked on a spot before abandoning |
| `PARKING_SCAN_INTERVAL` | `3` | Steps between full street scans (Scenario A) |
| `PLOTTER_UPDATE_INTERVAL` | `5` | Steps between plotter refreshes |
| `DB_SYNC_INTERVAL` | `60` | Seconds between database syncs |
| `SCENARIO_A_NAME` | `"Baseline"` | Scenario A identifier in the database |
| `SCENARIO_B_NAME` | `"Smart_Booking_Priced"` | Scenario B identifier in the database |
| `STREET_SPOT_THRESHOLD` | `3` | Max capacity for street-level aggregation (Scenario B) |
| `WEIGHT_DISTANCE` | `1.0` | Distance multiplier in cost function (Scenario B) |
| `WEIGHT_PRICE` | `100.0` | Price multiplier in cost function (Scenario B) |

Provides real-time visualization of simulation metrics in a separate process using matplotlib.

### `core/parking_logic.py` — Street-Level Parking Search (Scenario A)

Simulates a realistic human driver scanning streets and intersections for vacant spots. The core principle: SUMO's `setParkingAreaStop` is the sole arbiter of whether parking is feasible — the module does not pre-judge distances.

| Function | Signature | Description |
|----------|-----------|-------------|
| `reroute_random()` | `(vid, all_edges, opposite_map, outgoing_map)` | Assigns a random edge as a new destination, excluding the current edge, its opposite, and directly adjacent edges to ensure routes are at least 2 hops. Returns `False` on failure. |
| `scan_street()` | `(vid, current_edge, current_lanepos, spots_by_edge, all_spots, opposite_map, outgoing_map, edge_lengths, full_scan)` | Scans the current street (and optionally cross-streets and next route edge) for vacant spots within sight distance. Returns `(spot_id, spot_edge)` or `(None, None)`. |
| `try_park()` | `(vid, spot_id, spot_edge, stats, current_edge, all_spots)` | Attempts to park at a found spot. Same-edge: calls `setParkingAreaStop`. Different edge: calls `changeTarget` and records a pending reservation. Refuses new spots if one is already committed. Returns `True` on success. |
| `check_pending()` | `(vid, stats, current_edge, all_spots, all_edges, opposite_map, outgoing_map)` | When a vehicle arrives at the edge containing its pending spot, attempts to finalize parking. Cleans up if the spot is now occupied. |
| `handle_occupied()` | `(vid, stats, current_edge, all_spots, all_edges, opposite_map, outgoing_map)` | Detects when a target spot becomes occupied or the vehicle leaves the target road. Releases the spot and triggers rerouting. |

### `run_scenario_A_baseline.py` — Scenario A: Blind Cruising

Implements the baseline scenario where vehicles have no reservation system. They are assigned random destinations and must scan streets for vacant spots.

**Internal functions:**

| Function | Description |
|----------|-------------|
| `_load_spots(cursor)` | Loads parking spot metadata from the database and parses `parking.add.xml` for geometric data (`startPos`, `lane`). |
| `_load_edges()` | Parses `demo.net.xml` to build a dictionary of road edges with from/to node coordinates and computed length. |
| `_build_opposite_map(all_edges)` | Builds a bidirectional mapping: for each edge, finds its opposite-direction counterpart. |
| `_build_outgoing_map(all_edges, opposite_map)` | Builds a mapping from each edge to its downstream edges (excluding self and opposite). |
| `_spots_by_edge(all_spots)` | Groups spot IDs by their parent edge for efficient street scanning. |
| `_init_stats(current_time)` | Creates the initial stats dictionary for a newly spawned vehicle. |
| `_settle(vid, stats, current_time, current_dist, spot_id, cursor, conn)` | Logs a successfully parked vehicle's results to the database. |
| `_settle_lost(vid, stats, current_time, cursor, conn)` | Logs a teleported/lost vehicle's results with `final_spot_id = NULL`. |
| `_process_vehicle(...)` | Per-step state machine for a single cruising vehicle: checks parking status, handles pending spots, scans streets, attempts parking, reroutes when needed. |
| `run_baseline()` | Main entry point. Orchestrates the full simulation lifecycle. |

### `run_scenario_B_smart.py` — Scenario B: Smart Booking & Dynamic Pricing

Implements the intelligent scenario where vehicles are assigned optimal parking spots at spawn time using a cost function, and prices adjust dynamically based on real-time occupancy.

**Internal functions:**

| Function | Description |
|----------|-------------|
| `_load_spots(cursor)` | Loads parking spot data with pricing information from the database. |
| `_compute_positions(all_spots)` | Retrieves the physical coordinates of each parking spot lane (requires TraCI to be running). |
| `_compute_pricing(all_spots)` | Calculates surge pricing every simulation step. Small street spots (capacity ≤ 3) are aggregated per street for occupancy calculation. Pricing tiers: >90% → 2×, >70% → 1.5×, else base. |
| `_find_best_spot(vehicle_pos, all_spots)` | Selects the optimal parking spot by minimizing `cost = distance × WEIGHT_DISTANCE + current_price × WEIGHT_PRICE`. Only considers spots with available capacity. |
| `_assign_vehicle(vid, spot_id, all_spots, veh_stats, current_time)` | Routes a vehicle to its assigned spot via `changeTarget` and `setParkingAreaStop`. Only increments the booking counter after all TraCI calls succeed (atomicity). |
| `_settle(vid, stats, current_time, spot_id, cursor, conn)` | Logs a vehicle's results to the `Cruising_Logs` table. |
| `_handle_departed(departed, all_spots, veh_stats, current_time)` | Processes newly spawned vehicles: configures driving parameters, finds the best spot, and assigns it. Logs a warning on assignment failure. |
| `_process_driving(veh_stats, sub_results, current_time, all_spots, cursor, conn, gui)` | Per-step processing of all driving vehicles: updates speed/distance/fuel metrics, detects teleportation, enforces target timeout (releases stuck vehicles), and detects successful parking. |
| `run_smart_booking_with_pricing()` | Main entry point. Orchestrates the full simulation lifecycle. |
